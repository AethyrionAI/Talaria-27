# PART A — #178a: the App Store validation blocker

## The defect

Build warning, Xcode-beta4:

    The CFBundleShortVersionString of an app extension ('1.0') must match that of its
    containing parent app ('1.0.0').

A warning at build time and a **hard rejection at App Store validation**. This must land before
any submission attempt.

## Find it, do not assume it

`project.yml` has `MARKETING_VERSION: "1.0.0"` at lines ~95, ~388, ~437 and a
`CFBundleShortVersionString: "1.0.0"` at ~113. So the `1.0` is coming from somewhere those do not
cover — a target with its own `info.properties` hard-coding `1.0`, or an extension target that
inherits a default because it sets neither.

**Locate the actual source before editing:** `grep -n '1\.0' project.yml` and check every
extension target (`TalariaWidgets`, the share extension) for its own `info.properties` block.
Fix at the source — align the extension to `1.0.0`, or better, have it inherit `MARKETING_VERSION`
rather than hard-coding a literal that will drift again at the next version bump.

## Verification

`xcodegen generate`, then confirm the warning is gone from a clean build, and confirm the
generated plists agree:

    /usr/libexec/PlistBuddy -c "Print :CFBundleShortVersionString" <each generated Info.plist>

**After every `xcodegen generate`, verify `aps-environment: development` survived** in
`Talaria/Talaria.entitlements`. This is a standing trap in this project (#44/#48).

**The regen goes in its own commit**, separate from the `project.yml` edit.

# PART B — #172: the deliver picker's one-way door

## The defect

`TaskEditSheet.swift`, `TaskDeliverPicker`. Verified in source:

- `@State private var useFreeText = false` has exactly ONE write site — `Button("Custom…") { useFreeText = true }`
- `body` is `if platforms == nil || useFreeText { freeTextField } else { menuPicker }`
- `freeTextField` offers no control that sets it back

Tapping **Custom…** swaps the menu for a raw `TextField` permanently, for the life of the sheet.
Identical mechanism to #168a, in the field directly above the one just fixed.

## Why it is milder than #168a, and why it still matters

Milder: the deliver picker already preserves an off-list value as a marked `(custom)` row
(`isCustomValue` / `currentLabel`), so nothing becomes unverifiable the way #168a made the skills
picker's preservation contract unverifiable. The property is reachable; only the affordance is
one-way. Also, the value is a single token rather than a delimited list, so free text is a more
plausible terminal state.

Still matters: there is no way back to the server-driven platform list once you leave it. A user
who taps Custom… to inspect the field, or by accident, then hand-types the sheet's most
typo-sensitive value — `telegram:-100999:42` shapes live here (see #171's PATCH-diff proof) —
with no list to fall back on.

## Fix

`#168a`'s `SkillsFieldMode` is already a general shape: a mode value type with `showsPicker` /
`offersEditAsText` / `offersReturnToPicker`, each gated on whether a list is actually available.
**Reuse it if it generalises cleanly; mirror it if not.** Add a `USE LIST` control to the deliver
free-text field, rendered **only when `platforms != nil`** — offering a return to a list that
cannot open is a second dead end, which is the exact mistake #168a's fix was careful to avoid.

Mirror the existing control's styling: `MonoLabel`, size 8, `Design.Tracking.mono`,
`Design.Brand.accent`.

Roughly 10 lines plus one test suite mirroring `SkillsFieldModeTests`.

## Do this too — the third-instance audit

Two of two hand-rolled `useFreeText` escapes in this sheet shipped as one-way doors. That suggests
the shape is the problem, not the author. Run `grep -rn "useFreeText" Talaria` — it is the whole
audit. **If a third instance exists, report it; do not fix it in this lane** (that is how #172
itself came to exist, and the discipline was right).

# PART C — #61: the degenerate-card coverage gap

## Surface correction first — three sessions went to the wrong screen

The connected-mode Sessions drawer is **server-fed**: `SessionsHermesClient.listSessions` maps
`row.title` / `row.preview` straight from the Hermes sessions API into `HermesSessionInfo`. The
on-device card never touches it.

**#61 renders ONLY via `conversation.title` / `generatedPreview`** — `ChatScreen`'s own header and
`LocalChatBackend.sessionInfo`, which builds the standalone session list. **#61 can only be
verified in standalone mode.** Do not attempt to verify it against a paired host.

## Root cause — the mixed-card branch of `LocalIntelligenceService.conversationCard`

When guided generation returns a title but an EMPTY preview, the function pairs the generated
title with `fallback.preview`. With a non-empty user turn, `fallbackCard` sets that preview to the
assistant's FIRST line — exactly the line a lazy generated title echoes.

`degenerateCardReason` was written to catch this and has a gap:

- containment branch requires `shorter.count * 2 >= longer.count` (title must cover HALF the preview)
- prefix-echo branch requires `shorter.count >= 24`

So **a generated title of 12–23 characters that is a verbatim prefix of a preview more than twice
its length passes BOTH checks.**

Device evidence (standalone, whoGoesThere, `cbcc824`): title `"I can't create a haiku"` (22)
against preview `"I can't create a haiku directly, but here's a simple one:"` (~57).
22 ≥ 12 but 22×2 = 44 < 57, so containment misses; 22 < 24, so prefix echo misses.

## Fix

**An exact verbatim prefix needs no length ratio.** Either drop the `>= 24` floor on the
prefix-echo branch, or add a distinct exact-prefix branch with no length condition at all.

Prefer the distinct branch: it keeps the existing two checks' tuning intact and makes the new rule
legible in `degenerateCardReason`'s logging, which already names which guard tripped. That logging
is load-bearing — it is how this was diagnosed — so **whatever branch fires must be named in the
log line.**

## Fail-first test

Assert the exact device case fails the guard before the fix and passes after:
title `"I can't create a haiku"` / preview `"I can't create a haiku directly, but here's a simple one:"`.
Add boundary cases either side of both the 24-char floor and the 2× ratio, so a future tuning pass
cannot silently reopen the same gap.

# PART D — #137: the migration re-fires on reinstall

## The defect

Device sequence (whoGoesThere, fresh build): app DELETED, then reinstalled. Owen performed no
opt-in, and the migration re-fired — resurrecting the permission wall and **overriding a
deliberate opt-out** (sensor streaming had been turned OFF on purpose). Not a lab edge case: a
reinstall with a surviving Keychain pairing is an ordinary user path.

## Fix — both halves, approved by Owen 2026-07-23

1. **Move the migration done-stamp to share the PAIRING's lifetime** — store it in the Keychain
   alongside the credential, so a reinstall with a surviving pairing correctly declines to
   re-migrate. `UserDefaults` is the wrong lifetime: it dies with the app container while the
   pairing does not.
2. **Make `!hadPersistedSettings` mean OFF, not ON.** No settings blob is no evidence of consent.
   Defaulting to ON treats a stored credential as a proxy for user intent, which is precisely the
   inversion that produced the override.

## Fail-first test, no device needed

    migrateIfNeeded(isPaired: true, hadPersistedSettings: false)

against a clean `UserDefaults` must NOT enable health and location. Write it failing first.

## Note for the device pass, not for this lane

Pass (1) fresh-install now needs a harder setup: **revoke/disconnect FIRST** so the Keychain entry
goes, THEN delete, THEN install. Deleting alone no longer produces a fresh device — which is the
point of half (1). Record this in the PR so the device lane inherits it.
